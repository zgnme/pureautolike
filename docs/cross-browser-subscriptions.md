# Cross-Browser Subscription Plan

PureAutoLike can support Chrome, Firefox, and Safari builds from the same
codebase, but public distribution and paid access are separate concerns.

## Current Public State

- Chrome / Chromium public install:
  [Chrome Web Store](https://chromewebstore.google.com/detail/pureautolike/abamkpcdpihjpaomdpaklhifpfbobgmm).
- Firefox build: generated by `npm run build:firefox`, but not publicly listed
  on Mozilla Add-ons yet.
- Safari build: generated by `npm run build:safari`, but must be wrapped and
  shipped through the Safari Web Extension / App Store flow.
- License backend: deployed for free beta checks and future paid access.

## Can Subscription Sync Work Across Browsers?

Yes, but not by relying on a Chrome Web Store install alone.

Browser extension ids and stores are different across Chrome, Firefox, and
Safari. A Chrome Web Store install cannot automatically prove that the same
person is using a Safari or Firefox build. Cross-browser paid access needs a
server-side identity that is independent of the browser store, for example:

- an email-based account;
- a license key;
- a payment customer id linked to email;
- a signed one-time activation link.

The extension already sends `installation_id`, `extension_id`, `version`, and a
release `channel` to the backend. That is enough for beta gating and store/build
allowlists. It is not enough for cross-browser paid sync until the user can bind
each browser install to the same backend entitlement.

## Recommended Paid Model

Use one PureAutoLike account or license key for all browsers.

1. User installs from the official browser store for that browser.
2. User enters an email or license key in the extension popup.
3. Extension sends the local `installation_id`, browser `extension_id`, release
   `channel`, version, and email/license key to the license backend.
4. Backend checks one shared entitlement table.
5. If the entitlement is active, every allowed browser build returns paid access.

This keeps Chrome, Firefox, and Safari synchronized without depending on a
single browser store's payment system.

## Release Requirements Before Public Firefox/Safari Links

Do not advertise Firefox/Safari as public install channels until these are done:

- Firefox build is signed and published on Mozilla Add-ons or distributed as a
  signed self-hosted XPI.
- Safari build is packaged through Xcode and published through the App Store or
  TestFlight.
- `ALLOWED_EXTENSION_IDS` includes the final Firefox extension id and Safari
  extension id in addition to the Chrome Web Store id.
- The extension popup has a paid-account/license-key binding UI, if beta mode is
  disabled.
- Privacy policy discloses the account/license identifier used for paid access.

## Current Rule

For now, GitHub should advertise Chrome Web Store as the primary public install
path. Firefox and Safari can be mentioned as supported source/build targets, but
not as one-click public installation options.
